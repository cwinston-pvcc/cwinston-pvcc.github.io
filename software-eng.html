    Software Lifecycle: Development Phase
  • Requirements analysis
    		•  This phase specifies what services the proposed system will provide, identifies any conditions (time constraints, security, and so on) on those services, and defines how the outside world will interact with the system.
  • Design
    		•  This phase involves creating a plan for the construction of the proposed system. It is in the design phase that the internal structure of the software system is established. The result of the design phase is a detailed description of the software system's structure that can be converted into programs.
  • Implementation
    		•  This phase involves the actual writing of programs, creation of data files, and development of databases.
  • Testing
    		•  Software testing is the process of evaluating and verifying that a software product or application does what it is supposed to do. The benefits of testing include preventing bugs, reducing development costs and improving perofrmance.
    modular implementation a software design technique that emphasizes separating the funcionality of a program into independent, interchangeable modules
    Unified Modeling Language (UML) a standard notation for the modeling of real-world objects as a first step in developing an object-oriented design methodology
    use case a set of possible sequences of interactions between systems and users in a particular environment and related to a particular goal
    use case diagram a graphical depiction of a user's possible interactions with a system
    Data relationships:
  • one-to-one
    		•  a type of cardinality that refers to the relationship between two entities A and B in which one element of A may only be linked to one element of B, and vice versa
  • one-to-many
    		•  a type of cardinality that refers to the relationship between two entities A and B in which an element of A may be linked to many elements of B, but a member of B is linked to only one element of A
  • many-to-many
    		•  a type of cardinality that refers to the relationship between two entities A and B where A may contain a parent instance for which there are many children in B and vice versa
    design pattern a predeveloped model for solving a recurring problem in software design
    Software quality assurance (SQA) a process that assures that all software engineering processes, methods, and work items are monitored and comply with the quality control systems adopted by an organization
    Software testing:
  • glass box testing
    		•  testing in which the software tester is aware of the interior structure of the software and uses this knowledge when designing the test
  • black box testing
    		•  tests that do not rely on knowledge of the software's interior composition
  • beta testing
    		•  testing performed by real users of the software application in a real environment
  • alpha testing
    		•  testing performed to identify bugs before releasing the product to real users or to the public
    Documentation:
  • user documentation
    		•  This type of documentation is meant to explain the features of the software and describe how to use them. It is intended to be read by the user of the software and is therefore expressed in the terminology of the application.
  • system documentation
    		•  documentation that describes the software's internal composition so that the software can be maintained later in its life cycle
  • technical documentation
    		•  documentation that describes how a software system should be installed and serviced
    Intellectual property law:
  • software license
    		•  a document that states the rights of the developer and user of a piece of software
  • software disclaimer
    		•  the software developer's formal notice to the user of its software that the developer is denying any and all liability for any damages arising out of using the software as well as denying any implied warranties
  • best defenses against software liability claims
    		•  One of the best defenses against software liability claims is to apply sound software engineering principles during the software's development, to use a level of care compatible with the software's application, and to produce and maintain records that validate those endeavors.
